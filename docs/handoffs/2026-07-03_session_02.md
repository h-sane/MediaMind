## Architectural decisions

### WS auth via query param (not header)
`BaseHTTPMiddleware` in Starlette only intercepts HTTP scopes. WebSocket upgrade requests are a separate scope and **bypass the middleware entirely**. The browser `WebSocket` API also cannot send custom headers. Solution: the `/v1/progress` WS endpoint validates the token from `?token=` using `hmac.compare_digest`, closes with code 4401 on failure. Token-in-URL is acceptable for a localhost-only app with no intermediaries. A test explicitly asserts that a missing/wrong token is rejected.

### execute endpoint safety model
The execute endpoint is the only place M2–M4 touches user files. Three layered guards:
1. **Expected-count check**: client sends how many files it approved; server recomputes the actual trash set and rejects (409) if counts differ — prevents a stale UI from trashing more than the user saw.
2. **Zero-keeper check**: 422 if any group would be left with no keeper after execution.
3. **Staleness stat check**: each file is `stat()`'d before trashing; vanished or inaccessible files become error `ManifestEntry` objects, never silently skipped.
Files with `resolution=NULL` are structurally excluded from the trash set (the `get_trash_set` query only returns `resolution='trash'` rows).

### JobManager uses `app.state`, not a module global
The previous `_registry` module-global in `libraries.py` made parallel tests flaky (shared state across test runs). All new singletons go on `app.state` via the FastAPI lifespan handler, and are dependency-injected via `request.app.state`. `libraries.py` was refactored to match.

### TestClient must be used as context manager for lifespan
FastAPI/Starlette's `TestClient` only runs lifespan startup/shutdown when used with `with TestClient(...) as c:`. All test fixtures were updated to use the context-manager form. The `test_api.py` fixture now `yield`s from inside `with`, which is the canonical pattern.

### Thumbnail endpoint keyed by member ID, not path
`GET .../duplicates/files/{member_id}/thumbnail` looks up the path from `duplicate_members.id`. This confines readable paths to files that were part of a dedupe scan — a token holder cannot read arbitrary files by guessing paths.

### No React Router for navigation
Three views (`libraries`, `library`, `dedupe-review`) don't justify the react-router dependency (CLAUDE.md rule: every dependency must justify itself). Navigation is a Zustand `view` field with `navigate()` and `back()` actions. Revisit if views exceed ~5.

### Thumbnail object URL management
`useThumbnailUrl` uses `useState + useEffect` (not `useQuery`) so the object URL can be revoked on unmount. A `useQuery` approach would not have a built-in cleanup mechanism for `URL.revokeObjectURL`.

## Problems encountered and solutions

1. **`find_duplicates` cancel ambiguity**: the function returns `[]` for both "cancelled" and "no duplicates found". The job runner explicitly checks `ctx.cancelled()` after the return and sets `job.result = None` on cancel (vs the actual result dict on success). A test asserts this.

2. **Event loop closed warning in tests**: After `TestClient` context exits, the asyncio event loop closes. Job worker threads may still be running and call `loop.call_soon_threadsafe`, which raises `RuntimeError('Event loop is closed')`. Fixed by wrapping all `call_soon_threadsafe` calls in `try/except RuntimeError: pass` in `jobs.py`.

3. **Schema version test broke**: `test_db_opens_twice_idempotently` asserted `schema_version == "1"` but now it's `"2"` after migration. Fixed to compare against `str(SCHEMA_VERSION)` constant.

4. **`BaseHTTPMiddleware` bypassed by WebSocket**: Discovered during design review (Fable flag). Fixed by in-endpoint token validation. Test added to prevent regression.

5. **`TestClient` lifespan not triggered without context manager**: All existing tests used `TestClient(create_app())` without `with`. Since the new `create_app()` uses a lifespan for `app.state`, tests failed (no `registry` on state). Fixed by converting all fixtures to `with TestClient(...) as c: yield c`.

6. **`persist_scan` called with `DuplicateGroup` objects from `find_duplicates`**: The `store/duplicates.py` module avoids importing `core/dedupe.py` to prevent circular imports. The `persist_scan` function accepts `list` with a duck-type annotation and accesses `.match` and `.files` and `.is_best` on the objects. This works because the actual objects are always `DuplicateGroup` and `DuplicateFile` from `core/dedupe.py`.

## Assumptions made

- The WS `?token=` approach is acceptable for a localhost-only desktop app (no intermediaries, uvicorn access logs are local).
- Object URL memory "leak" for thumbnails is acceptable in a desktop app — volumes are small (one dedupe review session with a few hundred photos). A production-hardened version could add a LRU cache with explicit revocation.
- `find_duplicates` progress callback fires per-file; the 0.2s throttle in `JobContext.report_progress` prevents WS flooding on fast disks.
- The `ConfirmTrashDialog` triggers a dry-run automatically on open (not on a user click). This is intentional — the user should see the plan immediately when they open the dialog.
- The `useThumbnailUrl` hook fetches each thumbnail independently on mount. For very large review screens (100+ groups), this could cause a burst of requests. Acceptable at V1 scale; add request queuing in M5+ if needed.

## Model routing strategy (established this session)

For future sessions, use this pattern:
- **Fable** — spawn at the start of a new major phase (new milestone boundary) to review the handoff and plan. Used once this session, produced the entire B1–F8 plan.
- **Opus** — skip for now; Fable + Sonnet covers the need without the middle tier.
- **Sonnet (main thread)** — everything else: implementation, debugging, tests, file edits.

The Fable agent used ~42k tokens for one planning session and produced a plan that drove ~3,000 lines of correct, tested code. That ratio makes it worth it at milestone boundaries. Not worth it for individual feature decisions.
