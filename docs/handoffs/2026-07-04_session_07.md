## Architectural decisions

### D1: msg_type envelope (not type) in WS messages
`_job_to_msg` uses `msg_type: "job"` not `type: "job"` because `type` is a first-class field on the `JobSnapshot` (the job category: "dedupe", "faces", etc.). Colliding these two meanings caused the original bug. The frontend checks `msg.msg_type === 'job'` and then reads `msg.type` as the job category.

### D2: DB path update in organize route (not scanner)
Updating `files.path` inside the organize execute/undo routes (not during a rescan) is the right place because: (a) only the organize route knows which files it moved, (b) it avoids requiring a full rescan just to fix paths, and (c) it makes the undo operation self-contained.

### D3: Preserve rejections by content_hash + frame_no + person_id
Keying rejected pairs by `(content_hash, frame_no, person_id)` rather than by face ID is correct because face IDs are regenerated on every rescan (the `DELETE FROM faces` wipes them). The combination of content_hash (file identity), frame_no (position within file), and person_id (who was rejected) is stable across rescans.

### D4: Skip stale-faces pruning when seen_file_ids is empty
If phase 1 yields no file IDs (e.g., empty folder), we skip the DELETE to avoid wiping all faces from a valid prior scan. The guard `if seen_file_ids:` handles this.

## Important implementation notes

- **WS reconnect**: The frontend WS auto-reconnects with exponential backoff (500ms → 5s). On reconnect, the backend sends a snapshot of all non-terminal jobs so the UI resyncs without a page refresh.
- **Organize DB path update**: Only updates `files.path`. `duplicate_members.path` is NOT updated (dedupe results go stale after organize — rescan dedupe to refresh). This is acceptable because dedupe and organize are separate features.
- **Stale-faces pruning runs before phase 2**: This means if a file was moved externally AND the content changed, the old faces are pruned before new ones are detected. This is correct.
- **`extract_zip` removal**: The function remains in git history if needed. `_flatten_zip` (the private version) is still present and used for ZIP-mode provider downloads.
- **Icon ICNS**: `electron-builder` on macOS can auto-convert a PNG to ICNS. The placeholder will work for a Windows-only build.
