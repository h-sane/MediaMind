## Bugs found and fixed

### Bug 1 — undecodable files never actually reached `People/_unsorted`

**File:** `backend/src/mediamind/core/organize_plan.py`

`build_organize_plan` only queried files via `files JOIN faces`. But a
decode failure (`extract_file_faces` in `core/faces/engine.py`) always
yields `faces=[]`, and `persist_face_scan` only inserts a `faces` row per
*detected* face — so a file with `decoded_ok=0` can never have a `faces`
row, and the `if not decoded_ok: dest_folder = "_unsorted"` branch inside
the join loop was structurally unreachable. Undecodable files were silently
staying in place forever instead of routing to the visible `_unsorted`
holding area — a direct violation of the safety invariant "everything routes
somewhere."

This was masked by two things: an existing test manually inserted a `faces`
row on a `decoded_ok=False` file (something the real pipeline never does),
and a completely unwired legacy module, `core/organizer.py` (with its own
`route_media` function and its own passing tests), tested equivalent-looking
routing logic that the API doesn't actually call — `api/routes/organize.py`
uses `core/organize_plan.py`, not `core/organizer.py`. Confirmed via
`grep` that `core/organizer.py` and its helpers were referenced nowhere
outside their own test file and historical handoff docs.

**Fix:** added a second query in `build_organize_plan` for
`files WHERE decoded_ok = 0` not already covered by the join, routing them
to `_unsorted` directly. **Also removed the dead `core/organizer.py` module
and its test file** (`test_faces_pipeline.py`) and the now-unused
`cluster_media_faces`/`ClusterResult` from `core/faces/clustering.py` —
keeping dead code with its own green tests creates exactly this kind of
false confidence.

Caught by: the real-media E2E test (first run failed exactly here, with the
real corrupt-file injection), confirmed with a synthetic regression test
(`test_plan_routes_real_decode_failure_to_unsorted`).

### Bug 2 — rejecting a pending match could silently re-assign it anyway

**File:** `backend/src/mediamind/store/persons.py`, `persist_face_scan`

Session 07's fix 12 (preserve rejections across rescans) correctly stopped
re-creating a `pending_matches` row for an already-rejected
(content_hash, frame_no, person_id) triple — but it did this by skipping
the *entire* pending block, including the line that resets
`faces.person_id = NULL`. A face's `person_id` is set to the matched person
at INSERT time regardless; the pending block's job is to null it back out
pending review. Skipping that step for a rejected match left the face
**silently assigned to the exact person the user had rejected**, with no
review at all — worse than not having fix 12, and a direct violation of
safety rule 6 ("review before commit").

This only manifests in the specific (real) scenario where a file's `faces`
row gets pruned as stale (fix 13) and the file is later re-seen as "new" —
found via a regression test, not the real-media run (the real-media test
doesn't control clustering precisely enough to trigger a named-person
pending match deterministically).

**Fix:** restructured the branch so a rejected match still nulls
`person_id` (stays unassigned, correctly reflecting "not this person") but
skips only the `pending_matches` INSERT (no repeat prompt).

### Bug 3 — a second `/organize/undo` call re-applied the original move

**File:** `backend/src/mediamind/store/audit.py`, `last_undoable`

`record_action` inserts every action (including `kind="undo"` records) with
`undone=0`. `last_undoable`'s query (`undone=0 AND dry_run=0 AND ok=1`) then
matched the *undo action's own record* on a second call — so calling
`/organize/undo` twice would "undo the undo," silently moving files back
into `People/_unsorted` a second time instead of returning 404 "nothing to
undo." Caught by the real-media E2E test's final `assert res.status_code ==
404` check on a second undo call.

**Fix:** excluded `kind = 'undo'` rows from `last_undoable`'s query — an
undo record documents a completed reversal, not a pending one; there is no
redo feature.

## Architectural decisions

**D1 — undecodable-file routing is a second, independent query, not a
rewrite of the join loop.** Kept the existing `files JOIN faces` loop
unchanged (still correctly handles named/noise routing) and added a
separate `WHERE decoded_ok = 0` query for files that can never have a
`faces` row. Minimal, additive, doesn't risk the working paths.

**D2 — dead code was deleted outright, not deprecated.** `core/organizer.py`
had zero live callers (verified via repo-wide grep) beyond its own test and
historical handoff docs describing past milestone work. Per CLAUDE.md:
"If you are certain that something is unused, you can delete it completely."
Keeping it would have continued to mask exactly the kind of gap Bug 1 was.

**D3 — the real-media E2E test samples videos rather than running all
~129.** Real InsightFace CPU inference costs ~1s/image and ~15-20s/video
(15 sampled frames each) regardless of file size or duration — the cost is
per-video-processed, not per-byte. Running the full video set would take
~40 minutes per run; a repeatable ~6-7 minute run was worth far more this
session than exhaustive video coverage, since every video still goes
through the identical code path per file.

**D4 — the E2E test always copies to `tmp_path`, never runs in place.**
`test/` is untracked and organize/execute + duplicate-trash are real
filesystem mutations; a copy makes repeated, unattended runs safe and
gives a pristine baseline every time.

## Problems encountered and solutions

1. **First E2E run failed at the `_unsorted` assertion** — this was
   expected/desired: the run started before the organize_plan.py fix
   landed, so it correctly reproduced Bug 1 live against real data before
   the fix, then passed after.

2. **Second E2E run failed at the "second undo → 404" assertion** — found
   Bug 3 (undo-of-undo). Not anticipated going in; found by the test design
   itself (checking that a second undo has nothing left to reverse).

3. **`test_plan_routes_real_decode_failure_to_unsorted` initially failed**
   with 0 planned moves instead of 1 — the test only called `upsert_file`
   once with `decoded_ok=None` (phase-1 shape) before calling
   `persist_face_scan` directly, so `files.decoded_ok` was never actually
   set to `0`. Fixed by mirroring the real runner's two-phase upsert
   (`None` then the real result) in the test setup.

4. **Real InsightFace model already present** — `~/.insightface/models/
   buffalo_l` was already downloaded from earlier `prototype/sort_media.py`
   work, so the E2E test could use the real provider without a fresh
   275MB download. `ProviderManager(Path.home() / ".insightface", ...)`
   plus one `mark_installed()` call was sufficient — `extract_subdir =
   "models/buffalo_l"` lines up exactly with where InsightFace's own
   default lookup already finds it.

## Important implementation notes

- **The real-media E2E test is opt-in and skips gracefully** — it will not
  run (and will not fail) in any environment without both the `test/`
  folder and the InsightFace model present. It is not part of CI or the
  default `pytest` invocation's meaningful surface, exactly like the
  existing `integration` marker for the blank-frame InsightFace test.
- **Video sampling in the E2E test is about processing time, not
  correctness** — every video, large or small, goes through the identical
  `extract_file_faces` code path with the same 15-frame sample; the subset
  choice only affects how many videos get exercised in one run, not which
  code paths do.
- **The `_unsorted` fix and the rejected-match fix are independent** — the
  E2E test found the first; the second was found only by a synthetic
  regression test designed to probe the exact scenario (file re-seen as
  "new" after stale-row pruning). Neither would have caught the other.
