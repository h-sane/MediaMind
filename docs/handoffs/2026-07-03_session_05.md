## Architectural decisions

### MultiPersonFileOut includes sample_face_id per person option (D1)
Instead of requiring a separate API call for each person's thumbnail, `PersonOptionOut` includes `sample_face_id` (MIN(face.id) for that person in that file). The existing `GET /v1/libraries/{id}/faces/{face_id}/thumbnail` endpoint serves the thumbnail. This avoids a new endpoint and reuses the proven face thumbnail infrastructure.

### person_id=0 signals "clear" for route choices (D2)
Rather than a separate DELETE endpoint, `POST /route-choices` accepts `person_id=0` to clear a choice. This keeps the API surface minimal and allows mixed batches (some set, some clear). The backend branches on `choice.person_id == 0` to DELETE vs. upsert.

### AuditScreen is library-level, not app-global (D3)
The audit screen is reached via OrganizeScreen (which already knows the library ID). It's a child of the library in the nav tree, consistent with all other library-scoped views. No global audit view needed for M7.

## Problems encountered and solutions

1. **PowerShell heredoc syntax conflict**: git commit with multiline messages fails with PowerShell `$(cat <<'EOF'...)` syntax. Solution: used `git commit -m @'...'@` heredoc, and fell back to Bash shell for the actual commit. (Recurring issue in this project — always use Bash for git commits with multiline messages.)

2. **`persist_face_scan` owners array for multi-face files**: The seeding fixture in tests needs to track which `FileFaces` entry each face belongs to via the `owners` array. For a file with 2 faces, both entries in `owners` point to the same `FileFaces` index. Verified by checking test fixture carefully.
