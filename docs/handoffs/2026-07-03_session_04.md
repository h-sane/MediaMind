## Architectural decisions

### Organize plan does NOT include zero-face files (D1)
Files scanned with 0 detections (e.g., landscape photos) are NOT in the plan and stay in their original locations. Only files that appear in the `faces` table are moved. Rationale: the user's goal is sorting media BY PERSON — files with no person don't belong in a person's folder. Keeps the organize action conservative and predictable.

### Organize plan built fresh at execute time (D2)
`build_organize_plan` is called both at preview and at execute. No plan is persisted between the two calls. Rationale: the face scan results or person names might have changed between preview and execute; building fresh ensures consistency. This is safe because the plan is cheap (pure SQL) and idempotent.

### Undo reads manifest_entries, not undo_data (D3)
The `organize_actions.undo_data` column exists in the schema but is not used. Instead, undo reads `manifest_entries WHERE action='moved'` and reverses each move. This avoids duplication and makes undo authoritative: it undoes exactly what was recorded, not a plan.

### pending_for_named enabled at the call site, not as a default (D4)
`make_face_scan_runner(pending_for_named=True)` is set in `api/routes/scans.py`, not as the function default. The function default remains `False` for test isolation (tests that don't want pending matches don't get them without opting in). The test fixture for M6 API tests seeds the DB directly, bypassing the runner.

### OrganizeScreen uses POST /preview (not GET) (D5)
Per the implementation plan's API surface (`POST /organize/preview`). This allows a future body parameter (e.g., custom `target_rel`) without changing the route. Currently takes no body (empty POST).

## Problems encountered and solutions

1. **`DownloadFile` field name mismatch in tests**: used `files=[DownloadFile(size=0)]` but actual dataclass has `downloads=[]`, `filename`, `size_bytes`, `archive`, `extract_subdir`, `kind` fields. Fixed by using `kind="fake"` catalog entries (no `DownloadFile` needed; `is_installed` returns True automatically).

2. **`organize_plan` for multi-person files needs a per-file face count query**: when a file has multiple person clusters, we need to pick the dominant one. Rather than loading all faces into memory, we do a targeted `SELECT person_id, COUNT(*) FROM faces WHERE file_id=? GROUP BY person_id ORDER BY n DESC LIMIT 1` per ambiguous file. This is efficient since multi-person files are the minority.

## Important implementation notes

- **Dev venv**: `C:\Users\husai\faces-env` (Python 3.11.9). `app/.env` has `MAIN_VITE_PYTHON` pointing to it.
- **Run app**: `cd app && npm run dev`
- **Run backend only**: `cd backend && C:\Users\husai\faces-env\Scripts\python.exe -m mediamind`
- **Run tests**: `cd backend && C:\Users\husai\faces-env\Scripts\python.exe -m pytest -m "not integration"`
- **Organize places files in `<library_root>/People/`** (default `target_rel="People"`). Subfolder naming: `safe_folder_name(person.name or person.auto_label)`. Special folders: `_noise` (noise cluster), `_unsorted` (decode-failed files with face rows), `_other` (person with no display name — shouldn't happen in practice).
- **Undo only reverses `action='moved'` entries** from the most recent non-dry-run successful organize action. If the destination file has been moved/deleted since, `shutil.copy2` will fail and be recorded as an error in a new audit entry.
- **`pending_for_named=True`**: enabled for all face scans. New files where a named person is recognized will have `faces.person_id = NULL` and a `pending_matches` row. Existing files (already in `faces` for this provider) are not staged as pending.
- **`OrganizeScreen` fetches preview on mount** (React Query). The preview call is a POST with empty body to `/organize/preview`. If there's no face scan, it returns 422 and shows "run a face scan first".
- **`PersonsOut.pending_count`**: counts `SELECT COUNT(*) FROM pending_matches WHERE decision IS NULL`. Updated every time `list_persons` is called. Shown as amber badge in PeopleScreen.

## Known bugs / limitations

- **MultiPersonReview not implemented**: files with multiple people are auto-assigned to the dominant person at organize time. The `route_choices` table is wired in `organize_plan.py` but the UI to populate it doesn't exist yet.
- **Organize does not update the DB after moving files**: after organize, the `files.path` rows still point to the old locations. This is fine because the filesystem is the source of truth — a rescan after organize will update paths. But it means a second organize would try to move files from their old paths (which no longer exist) and fail. Users should rescan after organizing. This is acceptable for M6 but should be documented.
- **buffalo_l download not tested end-to-end** (same as M5 session — requires the ~275 MB download).
